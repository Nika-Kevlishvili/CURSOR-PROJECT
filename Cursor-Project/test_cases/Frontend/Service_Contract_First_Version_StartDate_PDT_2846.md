# Service Contract — First Version Start Date Re-alignment (PDT-2846)

**Jira:** PDT-2846 (Phoenix Delivery)  
**Type:** Task (Customer Feedback)  
**Summary:** Frontend verification that when a Service Contract version 1 is finalized with a past `signingDate`, the UI correctly reflects the auto-realigned `startDate`, version dropdown labels update, and form behavior is consistent.

**Scope:** Service Contract create/edit UI pages — version dropdown, date fields (signing date, start date), save/update behavior, version 1 vs non-first version UI differences, and post-save display after backend auto-realignment.

---

## Test data (preconditions)

Shared setup for all frontend test cases (environment + entity chain).

- **Environment:** Dev

1. Log into the portal with a user that has permissions to create and edit Service Contracts.
2. Create a customer via backend (`POST /customer`, type: PRIVATE, status: ACTIVE) or via the Customer UI page.
3. Create a service with status ACTIVE and at least one active version (via backend or Service management UI).
4. Navigate to **Service Contracts** → **Create new service contract**.
5. Fill in all required fields:
   - Select the customer from step 2.
   - Select the service from step 3 and its active version.
   - Set contract status: `Draft`, sub-status: `Draft`.
   - Set contract version status: `Draft`.
   - Leave signing date empty (required for Draft).
   - Fill in all required service parameters and additional parameters.
6. Save the contract. Verify creation succeeds.
7. Note the contract number and navigate to the contract's edit page.
8. Confirm: version 1 is displayed in the version dropdown, `startDate = today` (creation date), `signingDate` is empty.

## Frontend Test Cases

### TC-FE-1 (Positive): Version 1 update with past signingDate — UI reflects realigned startDate after save

**Description:** Verify that when a user updates version 1 with a signing date in the past and saves, the UI reloads and shows the new `startDate` equal to the `signingDate` (auto-realigned by backend).

**Preconditions:**
1. Apply Test data steps 1–8.
2. Delta: none (version 1 in Draft with startDate = today).

**Test steps:**
1. On the service contract edit page (version 1), change contract status to a finalized state (e.g. `Entered into force`) and contract version status to `Signed`.
2. Set **Signing date** to a date in the past (e.g. 7 days ago).
3. Set **Entry into force date** to today or in the past.
4. Leave the **Start date** field unchanged (it should show today and may be read-only for version 1).
5. Click **Save**.
6. After save completes, observe the contract detail page or reload.

**Expected test case results:** Save succeeds (no error toast). After reload, the `startDate` displayed on the page equals the signing date (7 days ago), not the original creation date. The version dropdown label shows the updated start date in the format: `1 / (signingDate - endDate)`.

### TC-FE-2 (Positive): Version dropdown label updates after realignment

**Description:** Verify that the version dropdown correctly reflects the new `startDate` after auto-realignment.

**Preconditions:**
1. Apply Test data steps 1–8.
2. Delta: Update version 1 with `signingDate` = 10 days ago, `contractStatus` = `Entered into force`, `contractVersionStatus` = `Signed`, `entryIntoForceDate` = today. Save successfully. After save, the backend auto-realigns version 1's `startDate` to 10 days ago.

**Test steps:**
1. After the save from the precondition, reload the page or navigate back to the contract's edit page.
2. Open the version dropdown at the top of the page.
3. Read the version 1 label text.
4. Verify the date portion of the label matches the realigned startDate.

**Expected test case results:** The version dropdown shows version 1 with the realigned start date in the label format: `1 / (DD.MM.YYYY - DD.MM.YYYY) Valid` where the first date equals the signingDate (10 days ago), not the original creation date. The label is generated by `create-edit-service-contract.component.ts` lines 1528-1544 using `formatDate(version.startDate)`.

### TC-FE-3 (Positive): Version 1 — start date field is read-only on edit (user cannot manually change it)

**Description:** Verify that when editing version 1, the start date field is not directly editable by the user (the modal for changing start date is skipped for version 1 updates per UI code).

**Preconditions:**
1. Apply Test data steps 1–8.
2. Delta: none (version 1 in Draft).

**Test steps:**
1. On the edit page for version 1, observe the start date field.
2. Attempt to change the start date directly (click on the field, try entering a new date).
3. Click Save (without changing contract status).

**Expected test case results:** The start date field for version 1 is read-only or the start date modal is not shown. The UI reuses the existing `startDate` for version 1 updates (per code: `this.createReqObject(existingVersion, object, this.currentVersion.startDate)`). The user cannot manually type a different start date for version 1.

### TC-FE-5 (Negative): Version 1 — signing date field empty for Draft status

**Description:** Verify that when version 1 is in Draft status, the signing date field is empty and the user cannot fill it (or if filled, save is rejected).

**Preconditions:**
1. Apply Test data steps 1–8.
2. Delta: Version 1 remains in Draft status; the negative trigger is attempting to fill the signing date field, which Draft status disallows.

**Test steps:**
1. On the edit page for version 1 (Draft status), observe the signing date field.
2. Attempt to enter a signing date (e.g. today).
3. Try to save without changing the contract status from Draft.

**Expected test case results:** The save fails with a validation error toast containing "Signing date should be empty" (the backend rejects non-null signingDate for DRAFT via `ServiceContractDateService`). If the UI proactively disables the signing date field for Draft status, the user cannot even enter a value — in that case verify the field is disabled/hidden. No startDate realignment occurs.

### TC-FE-6 (Positive): Version 2 — start date modal appears (unlike version 1)

**Description:** Verify that when creating a new version (version 2+), the start date modal/selection IS shown (unlike version 1 where it is skipped).

**Preconditions:**
1. Apply Test data steps 1–8.
2. Delta: Finalize version 1 (set to Signed with a signing date). Then trigger "Save as new version."

**Test steps:**
1. On the service contract edit page, click "Save as new version" (or equivalent UI action).
2. Observe whether a modal or form for selecting the new version's start date appears.

**Expected test case results:** A start date selection modal/form appears for the new version. The user can pick a start date for version 2 (unlike version 1 where the modal is skipped and existing startDate is reused). The proposed start date must be after version 1's startDate.

### TC-FE-7 (Positive): Post-realignment — signing date and start date consistent in the UI read view

**Description:** Verify that after realignment, the read/view mode of the contract shows consistent signing date and start date.

**Preconditions:**
1. Apply Test data steps 1–8.
2. Delta: Update version 1 with signingDate = 14 days ago, contractStatus = Entered into force, entryIntoForceDate = today. Save successfully.

**Test steps:**
1. Navigate to the contract's view page (read-only mode) via `GET /service-contract/{id}?versionId=1` or the UI view page.
2. Check the displayed signing date and start date.

**Expected test case results:** Signing date shows "14 days ago" date. Start date also shows "14 days ago" date (realigned). Both dates are consistent and clearly displayed. No stale data showing the original creation-date as startDate.

### TC-FE-8 (Negative): Version 1 — save with past signingDate but missing entryIntoForceDate for Entered-into-force — error toast

**Description:** Verify that the UI correctly shows a validation error when the user sets a past signing date but forgets to provide the required entry-into-force date for `Entered into force` status.

**Preconditions:**
1. Apply Test data steps 1–8.
2. Delta: none (version 1 in Draft or Ready).

**Test steps:**
1. On the edit page, change contract status to `Entered into force` and version status to `Signed`.
2. Set signing date to 7 days ago.
3. Leave entry-into-force date empty.
4. Click Save.

**Expected test case results:** Save fails. An error toast or inline validation message appears: "Entry in force is mandatory" (or similar). The contract is not saved. The startDate realignment does not matter because the overall update is rejected.

### TC-FE-9 (Positive): Multiple versions displayed with correct start dates after version 1 realignment

**Description:** Verify that when there are multiple versions and version 1's start date has been realigned, the version list/dropdown shows all versions with correct start dates in chronological order.

**Preconditions:**
1. Apply Test data steps 1–8.
2. Delta: Finalize version 1 with signingDate = 20 days ago (startDate realigns to 20 days ago). Create version 2 with startDate = 10 days ago. Both versions are in a valid state.

**Test steps:**
1. Navigate to the contract's edit page.
2. Open the version dropdown or version list.
3. Observe the ordering and date labels.

**Expected test case results:** Version dropdown shows:
- Version 1: `1 / (20 days ago - 10 days ago) Valid`
- Version 2: `2 / (10 days ago - open) Valid`
Versions are sorted by startDate. The dates reflect the realigned version 1 startDate, not the original creation date.

### TC-FE-10 (Negative): Version 1 — attempt to change startDate via form while contractStatus stays Draft — error

**Description:** Verify that the frontend form does not allow startDate modification for version 1 when no valid signingDate-based realignment is applicable (i.e., when contract remains in Draft).

**Preconditions:**
1. Apply Test data steps 1–8.
2. Delta: Version 1 remains in Draft status; the negative trigger is attempting to change `startDate` via the UI form, which the component should prevent for version 1.

**Test steps:**
1. On the edit page for version 1 (Draft status), note the startDate value.
2. If the startDate field allows editing (some UI configurations), change it to a different date.
3. Click Save (keep status as Draft, signingDate as null).

**Expected test case results:** The startDate field is not user-editable for version 1 — the UI skips the start date modal and reuses the existing `startDate` (per `create-edit-service-contract.component.ts` lines 1101–1104). If a direct API call with a different startDate were attempted, the backend would reject with "Start date must not be changed for this version." The startDate remains the original creation date.
