# Customer Invoices – Invoice date label translation key (PDT-2811)

**Jira:** PDT-2811 (Phoenix Delivery)
**Type:** Bug
**Summary:** The customer invoices list UI incorrectly reuses the translation key `billing_data_by_scales.date_of_invoice`, causing the column header and date range picker title to display 'Дата на фактура от мрежови оператор' instead of the expected 'Дата на фактура'.

**Scope:**
- **Affected area:** Customer → Edit customer → Customer Invoices tab (list view and list filters).
- **Affected UI elements:** (1) Invoice date column header in the invoices list grid; (2) Date range picker title in the list filters panel.
- **Expected behaviour after fix:** Both UI elements display 'Дата на фактура' in the Bulgarian locale.
- **Regression risk:** The Billing data by scales page uses the same translation key (`billing_data_by_scales.date_of_invoice`) and MUST still display 'Дата на фактура от мрежови оператор' — the fix must NOT alter that page's label.
- **Current wrong behaviour:** Both elements show 'Дата на фактура от мрежови оператор' (the network-operator-specific wording), which is incorrect in the customer invoices context.

> **⚠️ Mixed-state note (environment: Test):** Branch alignment to `origin/test` failed in the parent task due to a parser error. Test cases below are written from the cross-dependency context provided; local Phoenix UI source state may be partially stale. Testers should verify against the deployed Test environment.

## Test data (preconditions)

Shared setup — create all entities needed for a customer to have at least one invoice visible in the customer invoices list.

- **Environment:** Test
- **Locale:** Bulgarian (bg) — ensure the portal UI language is set to Bulgarian so translation keys resolve from `bg.json`.

1. Log in to the Test portal with a user account that has read and write permissions for Customer management and Billing (e.g. role: Administrator or Back-Office Operator).
2. Create a customer via `POST /customer` (type: PRIVATE, status: ACTIVE, customerIdentifier: auto-generated by the system).
3. Create a Point of Delivery (POD) via `POST /pod` (type: ELECTRICITY, status: ACTIVE, activation date: 2025-01-01). Note the POD identifier returned.
4. Create a product via `POST /product` (term: INDEFINITE, data delivery type: BY_PROFILE, status: ACTIVE). Note the product ID returned.
5. Create terms for the product via `POST /terms` (linked to the product from step 4; use default term settings).
6. Create a price component via `POST /price-component` (type: ENERGY, rate: 0.15 BGN/kWh, currency: BGN, linked to the product from step 4).
7. Create a product contract via `POST /product-contract` (linking customer from step 2, POD from step 3, product from step 4; status: ACTIVE, entry-into-force date: 2025-01-01, no termination date). Note the contract ID.
8. Create a billing run via `POST /billing-run` (type: STANDARD, billing period: 2025-01-01 to 2025-01-31, linked to the product contract from step 7).
9. Execute the billing run to completion — the run must generate at least one invoice with status GENERATED and amount > 0. Note the invoice ID returned.
10. Open the portal in a browser, navigate to **Customers → (search and open the customer from step 2) → Edit → Customer Invoices tab**. Confirm that the invoice from step 9 is visible in the customer invoices list.
