# Zero-amount liability and receivable prevention – All generation flows (PDT-2474)

**Jira:** PDT-2474 (Phoenix Delivery)  
**Type:** Customer Feedback  
**Summary:** Every flow that generates liabilities or receivables must prevent creation of entities with amount zero. The system has a global `ZeroAmountValidationListener` (`@PrePersist`) that throws `IllegalArgumentException` when `initialAmount == 0`. Individual flows must either skip creation or short-circuit before calling save, so that no transaction error occurs from the listener.

**Scope:** All liability generation flows (action, deposit, late payment fine, rescheduling, manual creation, payment, invoice from billing run / goods order / service order / disconnection, invoice cancellation, reversal of invoice / payment / manual liability offsetting) and all receivable generation flows (manual creation, deposit, invoice from billing run incl. credit notes / invoice corrections / VAT base adjustments / compensations, invoice cancellation, payments, reversal of payment / LPF / MLO / invoice / rescheduling). Each flow must gracefully handle zero-amount scenarios without producing zero-amount rows or unhandled exceptions.

---

## Test data (preconditions)

- **Environment:** Test
- **Customer:** Any active customer (legal or private; type does not affect zero-amount validation).
- **POD (Point of Delivery):** An active POD linked to the customer; identifier and type are not specific to zero-amount validation.
- **Product:** Any product with at least one price component; term and data delivery type are not relevant to this scope.
- **Product contract:** Status ACTIVE, entry-into-force date in the past, linked to the customer and a valid POD.
- **Billing run:** A standard billing run that has been executed for the contract (when invoice-related flows are tested).
- **Invoice:** Generated by a billing run or order; status depends on the specific flow under test.
- **Key services (from codebase):**
  - `CustomerLiabilityService` — central service for liability creation across all flows.
  - `CustomerReceivableService` — central service for receivable creation across all flows.
  - `ZeroAmountValidationListener` — `@PrePersist` JPA listener on both `CustomerLiability` and `CustomerReceivable` entities; throws `IllegalArgumentException` if `initialAmount == 0`.
  - `BillingRunStartAccountingInvokeService` — generates liabilities and receivables from billing run invoices.
  - `DepositService`, `PaymentService`, `LatePaymentFineService`, `ReschedulingService`, `ManualLiabilityOffsettingService`, `InvoiceCancellationService`, `InvoiceReversalProcessService`, `GoodsOrderProcessService`, `ServiceOrderProcessService`, `CompensationReceivableService`.

### TC-BE-1 (Positive): Manual liability creation with valid non-zero amount

**Description:** Verify that creating a liability manually via the API with a valid positive amount succeeds and the liability is persisted correctly.

**Preconditions:**
1. An active customer exists in the system.
2. The user has permissions to create liabilities.
3. The `CustomerLiabilityRequest` payload has `initialAmount` set to a positive value (e.g. 100.00 BGN).

**Test steps:**
1. Send `POST /customer-liability` with a valid payload where `initialAmount = 100.00`.
2. Inspect the response status and body.
3. Query the database to confirm the `customer_liabilities` row exists with the correct `initial_amount`.

**Expected test case results:** The API returns HTTP 200/201 with the created liability. The `initial_amount` in the database matches the request. No validation error occurs.
